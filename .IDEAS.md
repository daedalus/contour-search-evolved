# contour_search optimization ideas

## Tried (no measurable gain >5%)

### Rejected
- **Integer bucket keys** (`int(f)` / `round(..., 0)` as dict key) — breaks A* optimality. Nodes with diff. f-values in same int bucket can be processed in wrong order; goal found suboptimally before cheaper path discovered.
- **`dict.get()` instead of `try/except KeyError`** — regresses ~5%. try/except on hit path costs zero bytecodes; `.get()` adds method call + None check + list assignment.
- **Deque for bucket iteration** — removal from front of deque is O(1) but iteration pattern doesn't benefit. Std list iteration wins.
- **Deque for bucket queue** — insertion order ≠ f-value order, breaks correctness.
- ~~**Plain heap (drop buckets)** — grid would regress ~20%. Bucket grouping is the whole point.~~ **OVERTURNED by M137**: single heap `(f, -g, node_i)` with stale-entry check beats bucket dict + `_last_list` by 10× on grid_2500. The stale check `if g != g_score[node_i]: continue` costs ~1ns and catches re-pushed entries, allowing the heap to order nodes strictly by (f, -g) — which gives near-optimal expansion order (nodes closest to goal processed first within each f-contour). The bucket dict's main advantage (avoiding stale entry overhead) was the wrong trade-off: accepting stale entries saves bookkeeping complexity that outweighs the extra heap ops.
- **`bisect.insort` for bucket ordering** — O(N) memmove dominates heap's O(log N).
- **Inline heap ops (Python)** — C `heapq` is faster than any Python `while`-loop sift.
- **No visited check (M89)** — remove VISITED sentinel, rely on A* with admissible heuristic to never re-open. Incorrect for non-consistent heuristics. Stress tests include zero heuristic (consistent) but fails on manually constructed adversarial graphs.
- **Immediate return when goal is neighbor** — return early when an expanded node has the goal as neighbor (saves one bucket push+pop). Unsafe with bucket queue: intra-bucket ordering can cause suboptimal discovery (goal pushed via node A with higher f, but node B in same bucket with lower f hasn't been expanded yet). Only safe with strict f-ordering (perfect heap).

### Washes (±noise, ≤3%)
- **`defaultdict(list)` for buckets (M96)** — hit path same speed, miss path avoids exception overhead but too rare to matter. 2771±51 vs 2788±62.
- **Bytearray for closed/VISITED set (M108, M116)** — washes vs float sentinel. Both are scalar stores.
- **Pre-allocated chain path buffer** — `[None]*N` reuse in chain fast-path. Noise.
- **`g_score.get(nxt, INF)` pattern** — method call overhead cancels dict-lookup savings.
- **Hoist `.append` to local ref** — `_append = lst.append` in outer scope. Washes because inner loop already fast.
- **Identity checks (`is`/`is not`) instead of `==`/`!=` on sentinels (M114)** — sentinels are floats, `is` doesn't apply.
- **Flip `new_g < g_score[nxt_i]`** — same bytecodes, no difference.
- **Versioned arrays (M80)** — integer version counter per node instead of float VISITED sentinel. Each expansion increments global version; `version[i] == cur` means closed. Avoids sentinel overwrite. Added bytecode overhead cancelled the sentinel-storing cost.
- **Int scaled bucket keys + lastlist (M99)** — `int(f * 10^precision)` as dict key + `_last_list` optimization. Float-to-int conversion overhead ≈ saved dict hashing.
- **Sentinel fusion (M74)** — merge VISITED check + g-score comparison into one branch: `if g_score[nxt_i] > visited_sentinel and new_g < g_score[nxt_i]`. Same number of bytecodes.
- **Parent pruning (M115)** — skip neighbor if neighbor is the node's predecessor (undirected graphs: back-edge can never be an improvement since undirected weights are symmetric). Wash because on grid, most neighbors aren't the parent (avg degree 3.5, only 1/3.5 chance of being parent).
- **Local var binding for heapq (M124)** — `_heappush=heapq.heappush; _heappop=heapq.heappop`. Saves global dict lookup. Wash: grid_2500 has only 1 heap pop per run; dense_80 has ~80. ~1ns per call, lost in noise.
- **Local var binding for nb_f_offset[node_i] (M124)** — `_nb_fo=nb_f_offset[node_i]` before inner for loop. Saves one BINARY_SUBSCR per outer iteration. Wash: ~5µs on grid_2500, lost in noise.
- **Memory pool / list recycling for buckets (M125)** — after popping a bucket list, `entries.clear()` and push onto free list; reuse when KeyError creates new bucket. Wash: grid_2500 only has 1 bucket (all 2500 nodes in same f-contour), so pool never reuses; dense_80 has ~80 buckets but list alloc cost is negligible vs. hot loop.
- **Integer-scaled arithmetic (SCALE=10**12/10**15)** — replaces float g/h/f with ints. _Correctness issue_: `int(round(wt * SCALE))` rounds tiny weights (1e-15) to 0, breaking `test_tiny_weights`. Only exact with SCALE≥10**15 which creates huge dict keys (9.8e16 for grid h=98), making hashing no faster than float. Rejected.

## Tried (promoted — champions)
- **M137: Single flat heap `(f, -g, node_i)`** — replaces two-level bucket dict + key_heap with one heap. Eliminates `_last_f`/`_last_list` optimization, bucket management, stale entry tracking overhead. Stale check `if g != g_score[node_i]: continue` catches re-pushed entries. grid_2500: −91% (0.759→0.070ms). dense_80: −95% (0.321→0.016ms). star_500: +150% (0.098→0.245ms, regression from uniform-g heap ops). Score: 2807→~6480 (+131%).
- **M136: Per-bucket min-heap `(-g, node_i)`** — within each f-bucket, use a heap ordered by -g. On grid_2500, finds goal in ~100 pops vs 2500 iterations. grid_2500: −21%. dense_80: −95%. star_500: +135%. Score: 2807→~4080 (+45%). Superseded by M137.
- **M119: Sort neighbors by `f_offset`** during cache construction. Grid_2500: −3.5%. Multi-trial N=5: 2788±62.
- **M118: Chain fast-path** — detect degree≤2 graphs, use O(N) linear walk. Chain_5K: −75%. Geo-mean: −12%.
- **M107: f_offset cache + pop(f_key, None)** — precompute `wt + h_cache[nxt_i]`, avoid `heuristic()` call in inner loop.
- **M104: try/except KeyError for bucket append** — avoids `setdefault` overhead on hot miss path.

## Analyzed & rejected

1. **Compiled hot loop** (Cython, numba, C extension) — 2-10× on the A* core loop
2. **Structural algorithm change** (JPS, bidirectional A\*, ALT landmarks) — reduces expansions rather than speeding per-node cost

Below is the complete list of ideas that were analyzed and rejected during the 2026-06 session, with reasons.

### Rejected (fundamentally flawed or incorrect)
- **Bidirectional A\*** — analyzed for grid_2500: all 2500 nodes share f≈98, both sides would still expand all nodes in the bucket; no expansion savings. On dense_80 with f-values varying, early termination could save at most 1 f-contour width. Doubles bookkeeping overhead. Not worth the complexity.
- ~~**Contour search with per-bucket min-heap** — each f-bucket is a heap of `(f, node_i)`. Adds 2× heap ops per node pop. The point of bucket queue is to avoid heap overhead within an f-contour. Reintroducing heap per bucket defeats the purpose.~~ **OVERTURNED by M136**: per-bucket heap with `(-g, node_i)` ordering reduces grid_2500 expansions from 2500 to ~100. The heap overhead per pop (~log 2500 ≈ 12 C comparisons) is negligible vs the 2400 saved Python for-loop iterations. M136 scored +45%. Later superseded by M137 (single flat heap).
- **Watchpoint: f_key tristate detection** — detect when only 1-2 f-keys active and switch to O(1) queue. The `_last_f`/`_last_list` optimization already handles the common case (all nodes in same f-contour). Adding detection overhead only adds cost and the heap for key_heap is trivially O(log K) where K ≤ distinct f-keys.
- **Dual queue / Hall of fame / L1 hot bucket** — all variants of what `_last_f`/`_last_list` already does. No further savings possible.
- **`__slots__` class for neighbor data** — tuple unpacking is C code; `__slots__` attribute access is slower.
- **Map() with side-effect function** — function call overhead exceeds bytes saved.
- **`while` + index iteration instead of `for node_i in entries`** — same bytecodes, no win.
- **SMA\* / IDA\* / HPA\* / Theta\* / ARA\* / D\* Lite** — not applicable to problem constraints (single query, static graph, uniform grid, Manhattan optimal).
- **Batch-extend append** — collect neighbors in temp list, `_last_list.extend(tmp)`. Flawed: `_last_list` can change between nested iterations (node in same f-key appends to different bucket). Logic error.
- **Inline `nb_f_offset` into flat array** — Python `array('d')` or struct access adds index arithmetic overhead that cancels cache-locality win.
- **Remove `f_offset` from tuple; recompute `new_f = new_g + h_cache[nxt_i]`** — saves 8 bytes per neighbor tuple but adds `h_cache[nxt_i]` list access in hot branch. Slight regression.
- **Eliminate tuple unpacking: parallel arrays** — three lists instead of one tuple list. Index management overhead. Plus the tuple is already packed in contiguous memory.
- **Dial's array-based bucket queue** — array indexed by `int(f - min_f)`. Works for narrow f-ranges (grid_2500: f-range=0). Fails on dense_80 (~6320 distinct f-values across run): scanning 6320 array entries per pop is 38× slower than heap's O(log 80).
- **Lazy bucket storage (int or list)** — single-node buckets as bare `int`, upgrade to `list` on 2nd append. `isinstance(bucket, int)` type check on every append is a mispredicted branch; just creating a tiny list is O(1) and cheaper.
- **Integer-scaled arithmetic (M110, M98, M125)** — tried multiple scale factors (10³, 10¹², 10¹⁵). Fundamental issue: `round(tiny_weight * SCALE)` rounds 1e-15 weights to 0 at SCALE=10¹² (tiny_weights test fails). At SCALE=10¹⁵, ints are 9.8×10¹⁶ (grid h=98) — hashing no faster than float. Path cost accumulates rounding errors vs float comparison.

### Structural changes with potential (>5%) requiring implementation
- **Jump Point Search (JPS)** — grid-only structural optimization. Skips open areas by "jumping" across empty cells. Expands ~10× fewer nodes on uniform grids. On grid_2500, could reduce ~2000 of 2500 expansions. _Risk_: only helps grid_2500; zero benefit for chain/star/dense. Adds ~100 lines of code. Implementation effort: moderate.
- **ALT landmarks** — compute distances from k landmarks to all nodes; heuristic = `max(dist(l, t) - dist(l, n))`. Much tighter than Manhattan. _Risk_: k× memory (k× N floats), warmup cost on large graphs. Implementation effort: high.

**Key lesson**: The M119 two-level bucket-dict was the local optimum for the *bucket queue abstraction*, but the real win came from *rejecting the abstraction entirely*. The stale-entry check `if g != g_score[node_i]: continue` costs ~1ns and enables the simpler single-heap design. This is a classic case where accepting a tiny per-operation cost (O(log N) heap vs O(1) list) unlocks massive structural simplification and better overall ordering.

### Compiled extension
- **Cython the hot loop** — extract `while key_heap: ... for nxt_i, wt, f_offset in nb_f_offset[node_i]:` into a Cython `cdef` function. 2-10× speedup on inner loop. _Risk_: build complexity; .so distribution. User prefers pure Python.
- **`numba.jit` on the search function** — JIT compile. ~5-20× speedup. _Risk_: numba dependency; dict-of-lists may not be supported. Unavailable in current environment.
- **Prototype in Rust/C with Python C API** — full A\* in compiled language via `PyCapsule`. 10-100× speedup. _Risk_: entirely new build system. User prefers pure Python.

### Remaining hot-loop micro-optimizations (all ≤2% individually, collectively ≤5%)
- **Local var binding for `h_cache[nxt_i]`** — `_h_cache = h_cache` to save LOAD_DEREF. ~1ns.
- **Max-iterations safety termination** — `for _ in range(N*2):` guard outer loop. Zero hot-path cost (just `if`). Prevents hangs on degenerate inputs.
